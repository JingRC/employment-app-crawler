# 来源接口化实施路线图

## 目标

这份文档承接 [08_来源接口化分级.md](08_来源接口化分级.md)，不再讨论“适不适合”，而是明确：

1. 每个来源下一阶段要改哪些脚本。
2. 关键接口点在哪里。
3. 目前的阻塞项是什么。
4. 实施顺序应该怎么排。

当前重点放在三个最值得继续降浏览器依赖的来源：

1. 智联招聘
2. 猎聘
3. 前程无忧

## 总体优先级

建议按下面顺序推进：

1. 智联招聘
2. 猎聘
3. 前程无忧

排序依据：

1. 智联的真实列表接口已经定位到 search/positions，离 requests 直连最近。
2. 猎聘列表接口明确，但上下文与城市入口依赖比智联更重。
3. 前程无忧接口清晰，但 WAF 压力最明显，短期内更适合作为“浏览器辅助接口模式”的稳定化，而不是彻底去浏览器化。

1. search/positions 请求带动态签名，当前实现仍依赖浏览器运行态去拦截真实响应。
2. 目前脚本更像“首屏注水 + 翻页拦截”，而不是“requests 直接翻页”。

### 下一阶段改造动作

1. 继续扩大样本验证范围。

2. 在请求样本层补分页字段差异日志。
目标：比较第 2 页与第 3 页的 query/body 变化，确认真正决定翻页成功的字段集合。

3. 让 requests 分页分支支持连续多页复用。
目标：验证从第 2 页样本衍生到第 4、5 页时是否仍稳定，不稳定时再决定是否需要中途重新取样。

4. 给 run_incremental_update 增加更清晰的分支日志。
目标：让上层状态面板能区分 browser_page、request_page、request_fallback 三类分页来源。

### 目标状态

1. 首屏可继续保留 __INITIAL_STATE__。
2. 分页尽量收敛到 requests 直连 search/positions。
3. 浏览器退化为“签名初始化器”，而不是全程翻页执行器。

4. extract_footer_city_page_map
5. resolve_city_entry_via_footer
6. capture_current_page_payload

### 当前进展

1. 第一轮已完成：首屏和翻页拦截时会同步提取 pc-search-job 请求样本、headers、postData、Cookie。
2. 第一轮已完成：已增加 requests 复放探针，用于验证首屏样本能否脱离浏览器直连。
3. 当前验证结论：北京精确城市样本与全国样本的 requests 复放都未返回职位，说明猎聘暂时还不能直接沿用智联的分页 requests 化路径。
4. 诊断已补齐：当前全国样本日志已能看到 cookie 和 referer 都存在，但接口返回 status 200 且 items 为空，说明更像缺少隐藏上下文或额外 header，而不是直接被 WAF 页面拦截。
5. 第二轮诊断结论：当前“全国”任务中页面活动城市是全国，但首屏请求体里的 form_city 仍为 410，说明当前最值得优先核查的是页面上下文与请求体城市字段错位，而不是单纯补 Cookie。
6. 诊断已进一步收敛：在当前日志里，页面活动城市为全国、解析到的城市上下文码为空，但 mainSearchPcConditionForm.city/dq 仍保留 410，可以先把问题归到隐藏表单态残留，而不是 DOM 上的当前激活城市。
7. 初始化链路已补验证：当前 cond-init 请求已被捕获，但 request_city 为空，说明 410 不是由 cond-init 请求直接传入，更像后续 mainSearchPcConditionForm 组装时从页面隐藏状态继承而来。
8. 页面态补诊断后，当前全国样本里未捕获到隐藏字段、内嵌脚本片段或常见全局状态树里的 mainSearchPcConditionForm/city/dq，且浏览器主链路本身能正常抓到 6 条职位。现阶段要降低“410 本身就是错位根因”的权重，下一步更应检查浏览器真实请求对象里是否还有 requests 复放缺失的 header 或内部字段。
9. 全国 vs 北京 并排样本已确认：全国样本的 form_city=410，而北京精确检索样本会切到 form_city=010，说明 410 不是与城市无关的固定默认值，而是会随当前检索城市上下文变化。
10. 请求对象深挖已补到 extra_info：当前抓包里至少还能看到 associatedCookies、clientSecurityState、connectTiming、headers、requestId、siteHasCookieInOtherPartition 等浏览器侧附加字段；这些字段当前没有进入 requests 复放，已成为下一轮最值得优先补齐的缺口。

### 当前关键接口点

1. POST https://api-c.liepin.com/api/com.liepin.searchfront4c.pc-search-job
2. 页面城市筛选项 data-code/data-name
3. 页脚城市招聘网入口解析

### 当前阻塞项

1. 列表接口虽然清晰，但请求体依赖页面上下文与城市精确入口。
2. 不同城市入口可能对应不同页面状态与参数组合。
3. 当前“热门城市 + 更多城市页脚解析”的能力是在浏览器里完成的。

### 下一阶段改造动作

1. 继续记录更多 pc-search-job 请求样本差异。
目标：比较全国、热门城市、页脚城市入口三种上下文下 headers 与 body 的差异。

2. 补请求体发送方式和上下文字段诊断日志。
目标：确认当前空结果是由 Cookie、城市上下文、额外 header，还是接口风控导致。

3. 给 capture_current_page_payload 增加“城市上下文字典”输出。
目标：明确城市编码、query、分页字段、筛选字段之间的依赖关系。

4. 把 resolve_city_entry_via_footer 的输出从 URL 扩展成“城市请求上下文对象”。
建议字段：city_name、city_code、entry_url、query_params、headers seed。

5. 在确认上下文依赖前，不推进“第 3 页起优先 requests”分支。
目标：避免把尚未稳定的 requests 路径提前切入主链路。

### 目标状态

1. 浏览器负责城市入口解析与初始上下文建立。
2. 同城内多关键词与多页抓取尽量转给 requests。
3. 结果侧城市过滤只保留为异常兜底，不再是常态主链路。

### 当前脚本

1. job51_joblist_crawl.py

### 当前关键函数

1. fetch_city_code_map

3. extract_list_payload
4. extract_list_request_sample

### 当前关键接口点

1. 页面入口：https://we.51job.com/pc/search
2. 列表接口：https://we.51job.com/api/job/search-pc
3. 城市字典：https://js.51jobcdn.com/in/js/2023/dd/dd_city.json

### 当前阻塞项

1. 裸 requests 访问 search-pc 容易撞 WAF。
2. 当前虽然已经拦到了真实接口，但浏览器仍负责整个列表翻页链路。
3. 页面风控强于智联和猎聘，不适合一开始就追求完全 requests 化。

### 当前进展

1. 第一轮已完成：首屏和翻页拦截时会同步提取 search-pc 请求样本、headers 与 Cookie。
2. 第一轮已完成：已增加 requests 复放探针，用于验证浏览器初始化后的同会话直连可能性。
3. 当前验证结论：最小任务下 requests 复放返回 status 200，但响应为 text/html，并且正文前缀是设置 Cookie 的脚本页，说明当前主要阻塞就是 WAF/Cookie 初始化，而不是接口参数本身。
4. 第二轮实验已验证：把浏览器当前 Cookie 以 requests.Session 方式预热后，同一份首屏样本已经可以复放成功，说明前程无忧具备“浏览器预热一次，后续 requests 直连”的可行性。
5. 第三轮已验证：Java/北京 的最小实跑里，第 2 页和第 3 页已经命中“优先 requests、失败回退浏览器”的真实分页分支，并成功完成入库。
6. 扩测结论：Java/Python 在北京、上海、深圳的 5 页扩测中，首屏都要先走浏览器预热 Cookie；第 2、3 页已能稳定命中 requests 分支，但第 4、5 页在当前样本里还未稳定显示为 requests 直连，说明可以继续推广，但暂时不应把“深分页稳定 requests 化”视为完全完成。

### 下一阶段改造动作

1. 把 capture_current_page_payload 扩展成“浏览器初始化 + requests 尝试复放”双通路。
目标：先保留稳定性，再逐步测试 requests 边界。

2. 保存 search-pc 请求 headers、query 参数与 Cookie 样本。
目标：区分哪些字段是固定字段，哪些字段是浏览器态派生字段。

3. 新增一个 requests 试探函数，例如 replay_search_pc_page。
目标：在浏览器拿到 Cookie 后，直接请求第 2 页与第 3 页，看能否避免再次打开页面。

4. 把预热 Cookie 实验推进到第 2 页、第 3 页。
目标：验证预热一次后，分页 requests 是否还能继续稳定命中 JSON。

5. 视实验结果决定是否引入“第 2 页起优先 requests、失败回退浏览器”的分页分支。
目标：在不牺牲稳定性的前提下，逐步把浏览器从分页链路中移出。

6. 把 fetch_city_code_map 的输出缓存到运行态。
目标：避免每次任务都重新请求城市字典，提高后续 requests 方案的吞吐。

当前这两步已完成，下一阶段可以继续扩大到更多关键词、城市和第 4、5 页。
但当前更稳妥的说法是：前程无忧已经完成“首屏浏览器预热 + 中浅分页 requests 化”，深分页还需要继续观察。

### 目标状态

1. 浏览器负责初始化 Cookie 与过 WAF。
2. 成功初始化后，同一会话内分页尽量走 requests。
3. 如果 requests 重放失败，再退回浏览器拦截。

## 结论

下一阶段不是盲目追求所有来源“全 requests”，而是把浏览器从“全程执行器”逐步降级为：

1. 签名初始化器
2. Cookie 初始化器
3. 城市上下文解析器

三个来源的最终目标也不完全一样：

1. 智联优先争取分页 requests 化。
2. 猎聘优先争取同城上下文复用与多页 requests 化。
3. 前程无忧优先争取浏览器初始化后分页 requests 化，而不是完全脱离浏览器。

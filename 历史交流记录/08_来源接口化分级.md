# 来源接口化分级与建议

## 目标

这份文档用于回答一个实际工程问题：

当前接入的招聘来源里，哪些适合继续推进为纯接口爬虫，哪些更适合保留浏览器辅助链路，哪些只能把页面解析作为主要抓取方式。

这里的“纯接口可行”指的是：

1. 列表与详情接口已经明确。
2. 关键参数、城市映射、分页结构可稳定复用。
3. 不强依赖浏览器运行态、页面注水或人工验证。

这里的“浏览器辅助更稳”指的是：

1. 真实接口存在，但签名、Cookie、风控、WAF 或运行态依赖较重。
2. 通过浏览器打开页面、监听响应、读取注水数据后再入库，整体稳定性更高。

这里的“仅页面解析”指的是：

1. 当前没有稳定、可复用的真实列表接口入口。
2. 主要依赖页面 DOM、页面注水数据或结果页结构本身。

## 分组视图

### 纯接口可行

1. Boss直聘
原因：当前已经有稳定接口链路，适合继续保持 requests 主模式。

2. 实习僧
原因：真实搜索、详情、公司接口都已确认，后续进一步去浏览器化的收益很高。

### 浏览器辅助更稳

1. Boss直聘（浏览器拦截）
定位：Boss 主接口链路的补抓与回退模式。

2. 智联招聘
原因：翻页接口可见，但带动态签名，浏览器辅助拦截依然更稳。

3. 猎聘
原因：列表接口明确，但城市精确检索、页面入口和上下文状态仍依赖浏览器。

4. 前程无忧
原因：真实接口明确，但 WAF 明显，裸 requests 稳定性差。

5. 拉勾
原因：当前主要依赖页面注水与浏览器运行态；分页虽然补了 packet 和 next_data 解析，但仍不适合承诺为稳定纯接口来源。

### 仅页面解析

当前主来源里，没有一个来源建议长期停留在“仅页面解析”状态作为最终形态。

更准确地说：

1. 拉勾具备明显的页面解析成分，但当前实际落地是“浏览器辅助 + 注水解析 + DOM 回退”，不应简单归类为纯 DOM 爬虫。
2. 其余来源也都已经找到了不同程度的真实接口或可监听响应，因此不属于只剩页面解析一条路。

## 每个来源的推进建议

### Boss直聘

1. 保持 requests 主链路。
2. 浏览器拦截版本只作为 code=35/37、Cookie 失效或临时封控时的补抓通道。

### 实习僧

1. 继续把列表、详情、公司信息都收束到真实接口。
2. 浏览器仅保留在初始化 Cookie、校招特殊页面或异常回退场景。

### 智联招聘

1. 继续研究 search/positions 的签名参数来源。
2. 如果签名链可稳定还原，再考虑把翻页从“浏览器拦截”推进到“requests 直连”。
3. 在此之前，保持“首屏注水 + 翻页接口拦截”的混合模式最合理。

### 猎聘

1. 优先继续收紧城市精确检索入口解析，减少结果侧过滤。
2. 如果后续能稳定复用 pc-search-job 的上下文参数，再评估去浏览器化。
3. 当前不建议为了纯接口而牺牲城市精度和稳定性。

### 前程无忧

1. 保留浏览器拦截 search-pc 为主链路。
2. 可继续积累 WAF 前后的请求差异样本，但不要把“强行裸 requests”当近期目标。

### 拉勾

1. 当前最现实的方向不是“强上纯接口”，而是继续提高浏览器辅助链路的可观测性与恢复能力。
2. 首屏优先读取 __NEXT_DATA__，分页优先 next_data/packet，再回退 DOM，这条链路已经比单纯 DOM 稳定。
3. 后续若继续逆向，应重点围绕 f.j 这一职位列表查询分支，而不是泛化地抓所有网络请求。
4. 若站点继续维持滑动验证与路由注水模式，则应接受“人工验证检查点 + 自动继续抓取”的工程现实。

## 总体判断

如果把当前来源按“纯接口化优先级”排序，建议是：

1. 实习僧
2. Boss直聘主链路维护
3. 智联招聘
4. 猎聘
5. 前程无忧
6. 拉勾

这里的含义不是“拉勾最不重要”，而是“拉勾不适合优先投入到纯接口化改造”。

## 风控与验证的工程边界

对于滑块验证、验证码、人机校验，当前项目建议遵循下面的边界：

1. 不把自动绕过验证当成主方案。
2. 优先选择官方接口、授权合作、白名单或浏览器辅助模式。
3. 对强风控来源保留人工验证检查点，通过后继续自动抓取。
4. 把异常分支、快照、日志和历史任务信息补齐，比尝试脆弱的验证绕过更有长期价值。

## 结论

现阶段最合理的技术路线不是把所有来源都硬改成纯 requests 接口爬虫，而是分层处理：

1. 对 Boss直聘、实习僧继续推进纯接口能力。
2. 对智联、猎聘、前程无忧保持浏览器辅助接口模式，并持续降低浏览器依赖。
3. 对拉勾接受“浏览器辅助 + 注水解析 + DOM 回退 + 人工验证检查点”的现实约束，把精力优先放在稳定性、可观测性和恢复能力上。
